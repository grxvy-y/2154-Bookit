Problem Statement

Bookit · Group 48

The Problem

Community event organizers rely on manual, in-person registration. There is no way to:

  • Browse and purchase tickets online
  • Receive a digital ticket after purchase
  • Monitor real-time sales and capacity
  • Validate tickets at entry without manual checking

Stakeholders

Event Attendees

  • Access digital QR tickets

Event Organizers

  • Create & manage events
  • Monitor ticket sales in real-time
  • Enforce capacity limits

Venue Staff

  • Scan and validate QR tickets at entry — no manual list-checking

Solution & Architecture

Bookit · Group 48

Solution Overview

A responsive web application covering the complete ticket lifecycle — from creation to validated entry.

React 19 + Vite Tailwind CSS Supabase BaaS PostgreSQL + RLS qrcode.react html5-qrcode Vercel
  • Three user roles: Attendee, Organizer, Venue Staff
  • Row Level Security enforces data isolation at the database layer
  • Deployed to production on Vercel — auto-deploy from GitHub
Bookit · Group 48

Architecture

Presentation React + Vite · Tailwind CSS · React Router v6  ·  hosted on Vercel
Application Supabase Auth · Events & Orders · QR Validation · Organizer Dashboard · Realtime
Data PostgreSQL · 5 tables · handle_new_user trigger · RLS on all tables

Supabase BaaS eliminates a custom API server — auth, DB, and real-time out of the box.

Technical Highlights

RLS as Auth Layer

  • Access control enforced at the DB — not just the frontend
  • All 5 tables have RLS enabled
  • Role-based policies per table

QR Pipeline

  • crypto.randomUUID() assigned at checkout
  • QRCodeSVG renders ticket in-app
  • html5-qrcode reads camera feed
  • Validation marks ticket used atomically — race condition safe

Integration Challenges

  • Role assignment bug — Supabase Auth defaulting all users to Attendee; fixed by correcting the handle_new_user trigger
  • Dependency conflict — html5-qrcode incompatible with React 19; resolved via legacy-peer-deps in .npmrc
  • QR URL format — scanner rejecting full URL-encoded strings; fixed in ticketValidation.js
  • Capacity enforcement — overselling possible before atomic check was added at checkout

Lessons Learned & Future Work

Lessons Learned

  • BaaS reduces backend overhead significantly — but RLS complexity is non-trivial
  • Scope control matters — email descoping freed time for QR pipeline
  • Client-side UUID generation works but server-side signing would be more secure

Future Work

  • Attendee list export for organizers
  • Server-side signed QR tokens
  • Email confirmation via Resend
  • Refund portal
  • Waitlist functionality
